# The Toy POS System 🧸

ระบบบริหารจัดการจุดขาย (Point of Sale) และจัดการสต๊อกสินค้าที่ออกแบบและพัฒนาขึ้นมาเป็นพิเศษสำหรับร้าน **"The Toy"** โดยเน้นการรองรับโมเดลธุรกิจที่มี **"เจ้าของสินค้าหลายราย" (Multi-Vendors/Consignment)** และมีความซับซ้อนในการจัดการระบบการเงินรายวัน

---

## 🌟 ฟีเจอร์หลัก (Key Features)

### 1. ระบบจัดการหลายเจ้าของ (Multi-Owner Management)
- รองรับการฝากขายจากเจ้าของสินค้าหลายราย
- กำหนดอัตราส่วนแบ่งกำไร (GP Rate - Gross Profit) ให้กับเจ้าของแต่ละคนได้อิสระ
- สรุปยอดขาย, รายได้สุทธิ, และยอดเงินคงเหลือที่เบิกได้แบบแยกรายบุคคลแบบ Real-time
- **ระบบเบิกเงิน (Withdrawals):** บันทึกประวัติการเบิกเงินของเจ้าของสินค้าแต่ละราย และนำไปหักลบกับยอดรายได้สุทธิอัตโนมัติ

### 2. ระบบจัดการสต๊อกแบบแยกโซน (Dual-Zone Stock Management)
- **ตู้เก็บของ (Storage Qty):** สต๊อกหลักสำหรับรับสินค้าเข้า
- **ชั้นหน้าร้าน (Front Qty):** สต๊อกสำหรับวางขายจริง
- ฟังก์ชันการย้ายสต๊อกไปมาอย่างยืดหยุ่นด้วยระบบแสกนบาร์โค้ด:
  - รับของเข้าตู้ (Receive)
  - เติมของหน้าร้าน (Transfer to Front)
  - ดึงของกลับเข้าตู้ (Return to Storage)
  - ปรับลดยอด (Adjust/Reduce)

### 3. ระบบปิดยอดบัญชีประจำวัน (Daily Reconciliation)
เป็นหัวใจหลักของระบบ เพื่อให้การเงินโปร่งใสและตรวจสอบได้ 100%:
- **คำนวณยอดเงินที่ควรมี (Expected Revenue):** อ้างอิงจากยอดขาย - ส่วนลด
- **ระบุยอดเงินจริง (Actual Cash & Transfer):** พนักงานกรอกยอดเงินสดและเงินโอนที่นับได้จริง
- **จัดการส่วนต่าง (Difference Tracking):** 
  - หากเงินขาด/เงินเกิน ระบบจะบังคับให้ระบุ "สาเหตุ/หมายเหตุ" (Difference Note) เช่น ลูกค้าให้ทิป, ทอนเงินผิด
- **บันทึกของเสียและส่วนลด (Defects & Discounts):** แจกแจงให้เห็นชัดเจนว่าในวันนั้นมีมูลค่าของเสียและส่วนลดเท่าไหร่
- **ยอดยกไป (Carry Forward Cash):** ระบบนำยอดเงินทอนคงเหลือในเก๊ะ ยกยอดไปเป็นเงินทอนเริ่มต้นในวันถัดไปอัตโนมัติ

### 4. หน้าจอภาพรวมผู้บริหาร (Executive Dashboard)
- แสดงสรุปยอดขาย, ต้นทุน, กำไร, และค่าใช้จ่ายทั้งหมดในเดือนปัจจุบัน
- กราฟแสดงแนวโน้มการขายรายวัน
- การ์ดสรุปรายการส่วนต่างเงินสด ที่สามารถคลิกดูรายละเอียดเชิงลึก (รวมถึงของเสีย, ส่วนลด, และหมายเหตุ) ได้ทันที
- แจ้งเตือนสินค้าใกล้หมดสต๊อก

### 5. ระบบความปลอดภัยและบันทึกประวัติ (Security & Audit Trail)
- **ระบบ Audit Trail ขั้นสูง:** บันทึกทุกการเคลื่อนไหวสำคัญในระบบ เช่น การลบ/แก้ไขสินค้า, การจัดการเงิน, การสำรองฐานข้อมูล
- **บันทึกละเอียด (Data Comparison):** แสดงข้อมูล "ก่อนเปลี่ยน" และ "หลังเปลี่ยน" แบบ JSON เพื่อการตรวจสอบย้อนหลังที่แม่นยำ
- **ความปลอดภัย (Security Hardening):** 
    - บันทึกประวัติด้วยชื่อสิทธิ์ (Role Name) แทน ID เพื่อปกปิดโครงสร้างฐานข้อมูล
    - ติดตามการเข้าใช้งาน (Login/Logout Audit) ทุกครั้ง
    - ปกปิดข้อมูลละเอียดอ่อน (เช่น Password Hash) ในบันทึก Log

### 6. UI/UX สวยงาม ทันสมัย 
- รูปแบบปุ่มแบบโค้งมน (`btn-pill`) และมีเงา (`shadow-sm`) ทันระบบ
- ตารางข้อมูล (DataTables) ดีไซน์พรีเมียม (Premium Theme) อ่านง่าย ชัดเจน 
- ระบบแจ้งเตือนและยืนยันการทำงานผ่าน SweetAlert2 ที่ปรับแต่งปุ่มให้เข้ากับธีม

## 🚀 แผนพัฒนาฟีเจอร์การปิดยอดบัญชีรายเดือนในอนาคต (Monthly Settlement)

เพื่อรองรับการขยายตัวของร้านค้าเข้าสู่ระบบบัญชีมาตรฐาน ป้องกันปัญหางบการเงินย้อนหลังเปลี่ยนแปลง (Historical Data Modification) เมื่อมีการปรับเปลี่ยนอัตรา GP (Gross Profit) ของเจ้าของสินค้า หรือมีการทำรายการย้อนหลัง มีแนวทางการพัฒนาและข้อกำหนดการทำงานโดยละเอียดดังนี้:

### 1. วัตถุประสงค์และประโยชน์ (Objectives & Benefits)
* **ป้องกันงบการเงินย้อนหลังเปลี่ยนแปลง (Historical Data Protection):** เมื่อระบบบัญชีถูกล็อก (Settled) ข้อมูลการขาย ยอดเงินโอน และการตั้งค่าต่าง ๆ ในเดือนนั้นจะถือเป็นที่สิ้นสุด ไม่สามารถแก้ไข ยืนยัน หรือยกเลิกได้อีก
* **การใช้อัตรา GP ประวัติศาสตร์ (Historical GP Rate Freeze):** บันทึกค่า GP Rate ณ เวลาปิดยอดลงตาราง Settlements โดยตรง ช่วยให้อัตรา GP ที่อาจมีการเปลี่ยนแปลงในอนาคตไม่ย้อนกลับมาส่งผลกระทบต่องบการเงินย้อนหลัง
* **ความเป็นระบบและตรวจสอบได้ (Standard Payout Workflow):** มีขั้นตอนการปิดยอดและบันทึกสถานะการชำระเงินที่ชัดเจน (`pending` -> `paid`) พร้อมการเก็บหลักฐานการโอนและลงบันทึกในระบบประวัติการกระทำ (Audit Trail)

---

### 2. โครงสร้างฐานข้อมูลที่กำหนด (Database Schema Blueprint)
ตาราง `owner_monthly_settlements` ที่มีอยู่ในระบบจะถูกนำมาใช้งานจริงโดยมีรายละเอียดของแต่ละฟิลด์ดังนี้:

* **`id`** (int, PK): ไอดีหลักของรายการ
* **`owner_id`** (int, FK): อ้างอิงเจ้าของสินค้า (`owners.id`)
* **`settlement_month`** (varchar(7)): ระบุรอบเดือนในรูปแบบ `YYYY-MM` (เช่น `2026-05`)
* **`total_sales_amount`** (decimal(10,2)): ยอดขายสินค้าสะสมทั้งหมดของเจ้าของสินค้าในเดือนนั้น
* **`gp_rate_applied`** (decimal(5,2)): อัตรา GP ของเจ้าของที่นำมาใช้คำนวณจริง ณ เวลาปิดยอด
* **`gp_amount`** (decimal(10,2)): ยอดส่วนแบ่งที่ระบบดึงเข้าฝั่งร้านค้า (`total_sales_amount * (gp_rate_applied / 100)`)
* **`net_sales_amount`** (decimal(10,2)): รายได้ที่เจ้าของสินค้าควรได้รับก่อนหักเบิก (`total_sales_amount - gp_amount`)
* **`total_withdrawals`** (decimal(10,2)): ยอดเงินรวมที่เจ้าของได้กดเบิกไประหว่างเดือน (คำนวณจาก `owner_withdrawals`)
* **`net_payable_amount`** (decimal(10,2)): ยอดโอนเงินสุทธิสุดท้ายที่ร้านค้าต้องชำระจริง (`net_sales_amount - total_withdrawals`)
* **`status`** (enum('pending', 'paid')): สถานะการชำระเงิน
  * `pending`: ปิดยอดและล็อกงบเดือนนั้นแล้ว อยู่ระหว่างรอตรวจสอบและโอนเงินจริง
  * `paid`: ร้านค้าโอนเงินเรียบร้อยแล้วและปิดยอดสมบูรณ์ 100%
* **`created_by` / `updated_by`** (int): รหัสแอดมินผู้กระทำรายการ
* **`created_at` / `updated_at`** (datetime): วันเวลาที่มีการเพิ่มและอัปเดตข้อมูล

### 4. กฎการล็อกและควบคุมความปลอดภัยข้อมูล (Data Freezing Rules)
เพื่อควบคุมงบการเงินให้ได้มาตรฐานสูงสุด ระบบจะบังคับใช้กฎเหล็กทันทีเมื่อมีรายการปิดยอดประจำเดือนนั้นเกิดขึ้น (ไม่ว่าจะอยู่ในสถานะ `pending` หรือ `paid`):

1. **บล็อกแก้ไขรายการขาย (Sales Locking Rule):**
   * ระบบ API และส่วนการจัดการการขายจะไม่อนุญาตให้มีการเพิ่ม (Insert) รายการขายที่มีวันย้อนหลังในเดือนที่ปิดยอดไปแล้ว
   * บล็อกการแก้ไข (Update) หรือยกเลิก (Void) บิลการขายย้อนหลังใด ๆ ในเดือนนั้น
2. **บล็อกแก้ไขการเบิกเงินเจ้าของ (Withdrawal Locking Rule):**
   * ในสคริปต์ทำรายการเบิกเงิน (`dashboard_db.php`) หากแอดมินเลือกวันที่เบิกเงินย้อนหลังเข้าสู่เดือนที่ถูกปิดยอดแล้ว ระบบจะแจ้งเตือนและบล็อกไม่ให้เพิ่มข้อมูลลงตาราง `owner_withdrawals`
   * การยกเลิกรายการเบิกเงินเดิม (เปลี่ยนสถานะเป็น `void`) ที่เกิดขึ้นในเดือนที่ล็อกแล้ว จะถูกบล็อกการทำงานทันที
3. **การล็อกค่าคงที่ GP (GP Freezing Rule):**
   * ยอดขายจะถูกคำนวณและหักส่วนแบ่งเก็บลงตาราง `owner_monthly_settlements` ด้วย `gp_rate_applied` ในขณะปิดยอด ทำให้หากมีการเปลี่ยน GP Rate ของเจ้าของในอนาคต ยอดเงินประวัติศาสตร์จะไม่คลาดเคลื่อน

### 5. แนวทางการพัฒนาเชิงเทคนิค (Technical Implementation Steps)
* **Step 1: พัฒนาชุด API และ Helper ฟังก์ชันควบคุมการล็อกข้อมูล**
  * เขียนฟังก์ชันตรวจสอบการปิดยอด `isMonthSettled($conn, $dateString)` โดยรับค่าวันที่ใด ๆ และสืบค้นว่ารอบเดือนนั้นถูกบันทึกในตาราง `owner_monthly_settlements` หรือยัง
  * นำฟังก์ชันนี้ไปฝังในสคริปต์ `dashboard_db.php` ในส่วนคำสั่ง `save_withdrawal` และ `void_withdrawal` เพื่อดักและส่งข้อความ Error ปฏิเสธการทำงานหากมีการบันทึก/แก้ไขย้อนหลัง
* **Step 2: พัฒนาหน้าจอคำนวณยอดและรายงานปิดรอบ (Settlement UI)**
  * เพิ่มเมนูย่อย "ปิดรอบบัญชีรายเดือน" ใน `left-sidebar.php` ภายใต้หมวดหมู่รายงาน/บัญชี
  * ออกแบบ UI ที่สวยงามและพรีเมียมตามมาตรฐานของระบบ (ใช้ SweetAlert2 ในการยืนยันคำสั่ง และ Select2 ในการเลือกฟิลเตอร์เดือน/เจ้าของสินค้า)
  * แสดงยอดสรุปสะสมแยกตามเจ้าของสินค้า และปุ่ม "ยืนยันปิดยอด (Lock Month)"
* **Step 3: พัฒนาระบบประวัติและสถานะการโอนเงิน (Payout Management & Logs)**
  * หน้าตารางแสดงรายการเดือนที่ถูกล็อกทั้งหมดที่ค้างชำระ (`status = pending`)
  * ปุ่มยืนยันชำระเงินที่จะเปลี่ยนสถานะรอบเดือนเป็น `paid` พร้อมอัปเดตช่อง `updated_by` และบันทึก Log ลงในระบบ `audit_trails` เพื่อตรวจสอบย้อนหลังได้ทุกเวลา
